▼ Technical details (analyst)

Account Fixation Successful

The victim's browser session is now the attacker's KAYAK account.

Open kayak.com/profile/payment

The victim believes they are updating their own payment details — but this is the attacker's account. Any credit card entered here is stored in the attacker's account.

What happened

The victim clicked a link in a phishing email. That link opened this page, which sent a cross-origin postMessage to kayak.com/login containing the attacker's Google OAuth access_token.

KAYAK's postMessage listener checks event.data.action === "googleLogin" but does not check event.origin. It accepted the attacker's token from and called POST /k/run/googleauth/login — switching the victim's session to the attacker's account.

The Microsoft/Outlook OAuth handler in the same codebase correctly validates event.origin.

// Google — NO origin check (vulnerable): "googleLogin"===t.action ? (i&&i.close(), t.access_token ? n(t) : o(t)) : o(e) // Microsoft — HAS origin check (secure): if (e.origin !== expectedOrigin) return;
