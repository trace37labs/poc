KAYAK OAuth postMessage Account Fixation

CWE-346 ORIGIN VALIDATION ERROR
00
Vulnerability Overview
KAYAK's Google and Naver OAuth login flows accept postMessage events from any origin — no event.origin validation. An attacker page can inject its own OAuth token, silently logging the victim into the attacker's account.

Root Cause — Vulnerable vs. Secure Handler (same codebase)

// Google OAuth — NO origin check:
"googleLogin"===t.action ? (i&&i.close(), t.access_token ? n(t) : o(t)) : o(e)

// Naver OAuth — NO origin check:
"naverLogin"===t.action ? (o&&o.close(), t.access_token ? r(t) : n(t)) : n(e)

// Microsoft/Outlook — CORRECTLY validates origin:
if (e.origin !== expectedOrigin) return;  // ← missing above
What this PoC demonstrates: A cross-origin page (this page) opens kayak.com/login and injects an attacker-controlled Google access_token via postMessage. When the victim clicks "Sign in with Google", KAYAK processes the attacker's token instead — logging the victim into the attacker's KAYAK account.

What happened: This page (origin: ) sent a cross-origin postMessage to kayak.com/login containing the attacker's Google access_token. Because KAYAK's listener does not check event.origin, it accepted the token and called POST /k/run/googleauth/login — logging the victim's browser into the attacker's account.
Impact: Any credit cards, passport numbers, traveler details, or flight bookings the victim enters are stored in the attacker's account. The attacker logs in later to harvest this data. Affects all KAYAK regional domains (Google OAuth) and kayak.co.kr (Naver OAuth).

Fix — Add origin validation (consistent with existing Outlook handler)

window.addEventListener("message", function(event) {
    // Add this check — currently MISSING for Google and Naver:
    if (event.origin !== "https://accounts.google.com" &&
        event.origin !== window.location.origin) return;

    if (event.data.action === "googleLogin") {
        // ... existing token processing
    }
});
